## 二、核心概念：什么是 Embedding

Embedding = 把文本变成数字向量，让机器能"理解"语义。

`
"智云音箱 Pro"  →  [0.12, -0.34, 0.56, ..., 0.78]  (512 个数字)
"智能音箱产品"  →  [0.11, -0.33, 0.55, ..., 0.79]  (语义相近，向量接近)
"今天天气好"    →  [-0.45, 0.67, -0.12, ..., 0.34] (语义不同，向量远离)
`

**核心思想：语义相近的文本 → 向量距离近。**

## 三、Embedding 内部原理

### 3.1 整体架构

`
输入文本
    │
    │  ① Tokenization（分词）
    v
Token IDs
    │
    │  ② Embedding Lookup（查表）
    v
初始向量（每个 token 一个）
    │
    │  ③ Transformer Encoder（12层）
    │  Self-Attention + FeedForward
    v
上下文向量
    │
    │  ④ Mean Pooling（池化）
    v
最终句向量
`

### 3.2 第 1 步：Tokenization（分词）

把文本切成一个个 token（字/词）：

`
"智云音箱 Pro" → ["智", "云", "音", "箱", "P", "r", "o"]
`

每个 token 对应一个 ID（词表里的编号）：

`
字    ID      字    ID
智    3255    的    4638
云    756     是    3221
音    7509    在    1762
箱    5056    ...
`

词表是模型自带的「字典」，固定映射，训练好后不能改。BGE 的词表有 **21,128** 个 token。

### 3.3 第 2 步：Embedding Lookup（查表）

模型内部有一个巨大的**权重矩阵 W**，形状为 [词表大小, 向量维度]：

`
权重矩阵 W: [21128, 512]

查表 = W[ID] = 取第 ID 行 = 得到这个字的向量

W[3255] (智) → [0.15, -0.36, 0.58, 0.82, ...]  (512维)
W[756]  (云) → [0.18, -0.31, 0.52, 0.75, ...]
`

**权重矩阵就是模型本身**（占 366 MB），它是在海量文本上训练出来的「语义知识库」。

### 3.4 第 3 步：Transformer Encoder（注意力调整）

查表得到的初始向量**不知道上下文**——同一个字在任何地方向量都一样。需要通过 Self-Attention 调整：

`
"智云音箱" 中：
  "智" 关注 "云" (0.35) — 组成产品名
  "智" 关注 "音" (0.25)
  "智" 关注 "箱" (0.20)

新向量 = 0.35 × 向量_云 + 0.25 × 向量_音 + 0.20 × 向量_箱 + ...
`

经过 **12 层** Transformer 后，同一个字在不同上下文中向量完全不同。

**Transformer Encoder 每层包含 4 个组件：**

`
输入向量
    v
┌─────────────────────────────────────┐
│  1. Multi-Head Self-Attention        │  让每个字看到上下文
└──────────────┬──────────────────────┘
               v
┌─────────────────────────────────────┐
│  2. Add & Norm                      │  保持梯度稳定
└──────────────┬──────────────────────┘
               v
┌─────────────────────────────────────┐
│  3. Feed-Forward Network            │  提取更深层特征
└──────────────┬──────────────────────┘
               v
┌─────────────────────────────────────┐
│  4. Add & Norm                      │  防止信息丢失
└──────────────┬──────────────────────┘
               v
输出向量（传给下一层）
`

### 3.5 第 4 步：Mean Pooling（池化）

把多个 token 的向量合并成一个句向量：

`
向量 = (智 + 云 + 音 + 箱) / 4
`

最终输出一个固定长度的向量（如 512 维），就是这句话的「语义指纹」。

## 四、模型之间的区别

### 4.1 Embedding 模型 vs 生成模型

| 类型 | 架构 | 代表模型 | 能力 |
|------|------|----------|------|
| **Encoder-only** | 只有编码器 | BERT, BGE | 理解文本，不能生成 |
| **Decoder-only** | 只有解码器 | GPT, ChatGLM | 生成文本，不能做向量 |

### 4.2 为什么大模型不用编码层了？

`
2018: BERT (Encoder-only) = 只能理解，不能生成
2023: GPT (Decoder-only)  = 能理解，能生成
`

人类需求变了：「帮我写」> 「帮我理解」，所以 Decoder 赢了。

### 4.3 分离架构的好处

`
┌──────────────┐      ┌──────────────┐
│ Embedding    │ ---> │   LLM        │
│ (编码器)     │      │   (解码器)    │
│ BGE/OpenAI   │      │   GPT/GLM    │
└──────────────┘      └──────────────┘
  专注: 文本→向量       专注: 向量→回答
`

1. **各司其职**：编码的编码，生成的生成
2. **灵活替换**：想换 Embedding 模型？换就行
3. **成本降低**：Embedding 模型小，大模型专注生成
4. **效果更好**：每个模型都在自己的任务上优化到极致

### 4.4 向量维度对比

| 模型 | 维度 | 说明 |
|------|------|------|
| BGE-small-zh | 512 | 轻量，适合学习 |
| BGE-base-zh | 768 | 平衡性能和效果 |
| OpenAI text-embedding-3-small | 1536 | 效果最好 |
| BGE-M3 | 1024 | 多语言，支持 8192 token |

## 五、相似度计算

### 5.1 为什么要比较向量？

向量本身只是一串数字（语义指纹），需要通过**比较**才能知道哪个文档和问题相关。

`
文档1 向量: [0.12, -0.34, 0.56, ...]  (指纹)
文档2 向量: [0.11, -0.33, 0.55, ...]  (指纹)
用户问题向量: [0.25, -0.18, 0.41, ...] (指纹)

余弦相似度:
  问题 vs 文档1 = 0.85  ← 最相关！
  问题 vs 文档2 = 0.42
`

### 5.2 三种度量方式

| 度量方式 | 范围 | 最常用场景 |
|----------|------|-----------|
| **余弦相似度** | [-1, 1] | 文本/语义（最常用）|
| 欧氏距离 | [0, +∞) | 图像/数值 |
| 点积 | (-∞, +∞) | ML 模型内积 |

**为什么余弦相似度最常用？**
- 归一化后只看方向，不看大小
- "我喜欢猫" 和 "猫我很喜欢" 方向相同，余弦高

### 5.3 归一化

把向量缩放到单位长度（模为 1）：

`
归一化后：
- 点积 = 余弦相似度
- 所有向量都在超球面上
- 只看方向，不看大小
`

## 六、在 RAG 中的使用时机

### 6.1 两个阶段

| 阶段 | 做什么 | 用什么 |
|------|--------|--------|
| **添加知识库**（离线） | 文档→分段→向量→存储 | Embedding（生成向量）|
| **用户查询**（在线） | 问题→向量→比较→检索→生成 | Embedding + **余弦相似度** |

### 6.2 添加知识库时（离线，一次性）

`
文档 → DocumentLoader → TextSplitter → Embedding → 向量存储
`

只做 Embedding，生成向量存起来。不做比较，因为没有「问题」可比。

### 6.3 用户查询时（在线，每次）

`
用户问题 → Embedding → 问题向量
                          ↓
              余弦相似度（和所有文档向量比较）
                          ↓
              找出最相关的 Top-K 个
                          ↓
              喂给 LLM 生成回答
`

### 6.4 类比

- 添加知识库 = 给每个人拍照存档（只拍照，不比较）
- 用户查询 = 拿一张照片去匹配（这时候才比较）

就像公安系统：
- 采集指纹（离线）：只录入，不比较
- 破案比对（在线）：拿嫌疑人的指纹去比对

## 七、测试结果

### 7.1 相似度矩阵（5 个句子）

`
            音箱    摄像头    天气   灵犀模型   门锁
音箱       1.000   0.567   0.219   0.625   0.528
摄像头     0.567   1.000   0.252   0.402   0.588
天气       0.219   0.252   1.000   0.265   0.219
灵犀模型   0.625   0.402   0.265   1.000   0.309
门锁       0.528   0.588   0.219   0.309   1.000
`

**分析：**
- 最相似：音箱 vs 灵犀模型 (0.625) — 都是 AI 产品
- 最不相关：天气 vs 所有 (0.219-0.265) — 完全不同的语义
- 智云产品之间：0.5-0.6 — 同属一个产品线

### 7.2 性能数据

| 指标 | 数值 |
|------|------|
| 模型首次加载 | ~13 秒 |
| 单条编码延迟 | 7-12 ms |
| 模型大小 | ~100 MB |
| 向量维度 | 512 |

## 八、专业术语速查

| 术语 | 含义 | 用在哪 |
|------|------|--------|
| **Embedding** | 文本→向量的转换 | "用 Embedding 把文档向量化" |
| **Vector** | 数字向量，文本的数学表示 | "这个 chunk 的向量是 512 维" |
| **Dimension** | 向量维度，有多少个数字 | "BGE-small 是 512 维" |
| **Token** | 模型处理的最小单位（约 1.5 个中文字）| "这句话有 20 个 token" |
| **Vocabulary** | 词表，Token ID 到字的映射 | "BGE 的词表有 21,128 个 token" |
| **Weight Matrix** | 权重矩阵，模型的核心参数 | "权重矩阵就是模型本身" |
| **Encoder** | 编码器，把文本变成向量的模型 | "用 BGE Encoder 编码" |
| **Decoder** | 解码器，根据向量生成文本的模型 | "GPT 是 Decoder-only" |
| **Self-Attention** | 自注意力，让每个字看到上下文 | "Transformer 的核心机制" |
| **Feed-Forward** | 前馈神经网络，提取更深层特征 | "Attention 之后的 FFN" |
| **Pooling** | 池化，把多个向量合并成一个 | "Mean Pooling 取平均" |
| **Cosine Similarity** | 余弦相似度，衡量语义相似性 | "两个句子的余弦相似度是 0.8" |
| **Normalization** | 归一化，把向量缩放到单位长度 | "归一化后点积等于余弦" |
| **Top-K** | 检索最相关的 K 个结果 | "取 Top-5 最相关的文档" |
