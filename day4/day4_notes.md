## 二、向量存储原理

### 2.1 基本存储单元

向量数据库存储的核心是**稠密向量（Dense Vector）**，即 Embedding 模型把文本/图片映射成的浮点数数组。

`"音箱的价格是多少？"` → Embedding 模型 → `[0.12, -0.34, 0.56, ..., 0.78]`（512维或768维浮点数）

每个向量旁边还会存：
- **原始文本**：检索后返回给用户
- **元数据（metadata）**：来源、时间戳、标签等，用于过滤

### 2.2 向量库存储架构对比

不同向量库的存储方式各有侧重，核心区别在于持久化和扩展能力：

| 向量库 | 存储架构 | 持久化方式 | 特点 |
|--------|---------|-----------|------|
| **FAISS** | 纯内存索引 | 序列化到磁盘（`.index` 文件） | 最快，重启需重新加载 |
| **Chroma** | 嵌入式，SQLite + Parquet | 自动持久化到本地目录 | 零配置，开箱即用 |
| **Qdrant** | 自研存储引擎 | WAL + 快照，支持 S3 | 支持丰富的过滤条件 |
| **Milvus** | 分布式，存算分离 | 对象存储（MinIO/S3） | 海量数据，水平扩展 |
| **Pinecone** | 全托管云服务 | 自动管理 | 零运维，按量付费 |
| **Weaviate** | 自研，支持混合搜索 | LSM-tree 存储 | 原生支持向量+关键词混合 |

### 2.3 向量数据库可视化管理工具

向量数据库存的是高维向量，直接看数字很难理解数据分布。现在已经有成熟的桌面客户端，可以像 Navicat 管理 MySQL 一样，可视化地管理和探索向量数据库。

**[VectorDBZ](https://github.com/vectordbz/vectordbz)**（⭐238，MIT 协议）是一款开源的向量数据库桌面客户端，核心功能：

- 连接本地或云端向量数据库实例
- 浏览 Collections、查看数据和元数据
- 运行向量检索和混合检索
- **2D/3D 可视化 Embedding 分布**（直观看到向量聚类效果）

支持的向量库：Qdrant、Weaviate、Milvus、ChromaDB、Pinecone、pgvector、Elasticsearch、RedisSearch

适用场景：本地实验时可视化查看向量分布和检索结果，比命令行直观很多。搭配 Qdrant、Chroma 等有服务端的向量库使用效果最佳。

### 2.4 向量数据库内部存储机制

向量不是"按行存"的，也不是简单的 KV。各家向量库为了高效存储高维向量，设计了专门的存储引擎。

#### 核心结论

- 没有一家用传统关系型的"按行存"方式
- 向量和元数据**几乎都是分开存储**的（向量是固定长度 float 数组，和变长文本混在一起效率很低）
- 写入都用 **LSM-tree 或 WAL** 模式（追加写，后台合并），因为向量库的场景是"写入一次，读取很多次"
- 索引是**额外构建**的，不和原始数据混在一起

#### 向量存储形式总结

> **向量 = 固定长度 float 数组，紧凑排列在连续内存中，和文本/元数据分开存储，通过 ID 关联，索引单独构建。**

具体来说：

- **向量本体**：768 维 float32 向量 = 768 × 4 = 3KB，连续排列在内存数组中，支持随机访问
- **文本 + 元数据**：和向量分开存，通过整数 ID 关联（FAISS 返回 ID，你再去查对应文本）
- **索引（HNSW/IVF/PQ）**：根据向量数据额外构建的加速结构，不和原始数据混在一起
- **写入模式**：追加写（LSM-tree / WAL），后台合并，因为向量库是“写入一次，读取很多次”的场景
- **持久化**：FAISS 手动序列化 .index 文件，Chroma 自动存 SQLite + Parquet，Qdrant 用 WAL + 快照

#### FAISS — 纯内存数组

FAISS 不是数据库，是纯内存 C++ 库。没有持久化引擎，就是一个巨大的 float32 连续数组。

`
内存布局：
┌──────────────────────────────────────────┐
│ v1[0] v1[1] ... v1[511]  ← 向量1 (2048字节)
│ v2[0] v2[1] ... v2[511]  ← 向量2
│ ...                                       │
│ vN[0] vN[1] ... vN[511]  ← 向量N
└──────────────────────────────────────────┘
  索引直接存偏移量，搜索时直接读数组

文本和元数据：FAISS 自己不管！
  → 你需额外用 Python list/dict 存
  → 搜索返回 int ID，你自己去查对应文本
`

类比：像一个巨大的 Excel 表格，每行就是一个向量的原始浮点数。

#### Chroma — SQLite + Parquet 双层存储

`
持久化目录：
chroma/
├── chroma.sqlite3        ← SQLite：存元数据、文本、文档关系
└── <collection_id>/
    └── data_level0.bin   ← Parquet：存向量和索引

SQLite（关系型，管"这是什么文档"）：
┌──────┬────────┬──────────┐
│ id   │ text   │ metadata │
├──────┼────────┼──────────┤
│ "1"  │ "音箱.." │ {source:..}│
│ "2"  │ "价格.." │ {page: 3} │
└──────┴────────┴──────────┘

向量存储（自定义二进制格式，管"向量长什么样"）：
┌──────────────────────────┐
│ header (版本/维度/数量)     │
│ vector_id → HNSW 图结构   │
│ float32 数组（向量本体）    │
└──────────────────────────┘
`

#### Qdrant — Segment + WAL 列族存储

Qdrant 的设计最接近"专为向量设计的数据库"。

`
持久化目录：
storage/collections/my_collection/segments/
├── 0/
│   ├── vectors.bin      ← 向量数据（float32 数组）
│   ├── payload.bin      ← 元数据（JSON）
│   ├── graph.bin        ← HNSW 图结构
│   └── config.json
└── 1/ ...

每个 Segment 内部：
┌──────────────────────────────────────┐
│  Point（一条记录）= 主键 + 向量 + 元数据 │
│  {                                  │
│    id: 123,          ← 主键 (KV)     │
│    vector: [0.1,...], ← 浮点数组      │
│    payload: {                        │
│      text: "音箱...", ← 元数据        │
│      source: "manual"               │
│    }                                │
│  }                                  │
│                                      │
│  索引层：                              │
│  ├── HNSW 图索引（向量检索）           │
│  ├── payload 索引（字段过滤，倒排索引）  │
│  └── 全量扫描兜底                     │
└──────────────────────────────────────┘

写入流程：WAL → 内存 Buffer → 异步刷盘到 Segment → 定期 Compaction 合并
`

类比：像 HBase 的列族存储，向量一列、元数据一列、主键单独索引。

#### Milvus — LSM-tree + 对象存储

Milvus 架构最复杂，是分布式设计。

`
写入流程：
  Client → Proxy → Message Queue (Pulsar/Kafka)
                       ↓
                  Data Node（异步消费）
                       ↓
                  Segment（Growing → Sealed → Index）

Segment 结构：
┌────────────────────────────────────┐
│  Growing Segment（内存，可写入）     │
│  ┌─────┬───────────┬────────┬────┐ │
│  │ PK  │ Timestamp │ Vector │Field│ │
│  └─────┴───────────┴────────┴────┘ │
│            ↓ 刷盘                   │
│  Sealed Segment（不可写，建索引）     │
│  ├── Binlog 文件：行式存原始数据      │
│  ├── Log 文件：增量数据              │
│  └── Index 文件：ANN 索引            │
└────────────────────────────────────┘

持久化到对象存储（S3/MinIO），类似 Cassandra/TiKV
`

#### Weaviate — 双 LSM-tree

Weaviate 用经典 LSM-tree（和 RocksDB/LevelDB 同一思路），但把向量和属性拆成两棵独立的树。

`
写入流程：MemTable（内存有序）→ 满了刷盘 → SSTable → 定期 Compaction

Weaviate 存储结构：
┌──────────────────────────────────────┐
│  每个 Class（如 "Product"）一棵 LSM-tree │
│                                      │
│  ┌──────────┐  ┌──────────┐          │
│  │ 向量 LSM  │  │ 属性 LSM  │ ← 分开存 │
│  └──────────┘  └──────────┘          │
│                                      │
│  向量 LSM：key=doc_id, value=float32[]│
│  属性 LSM：key=doc_id, value=JSON    │
└──────────────────────────────────────┘
`

#### 各家存储模型对比

| 向量库 | 存储模型 | 类比 | 向量与元数据关系 |
|--------|---------|------|----------------|
| **FAISS** | 连续 float 数组 | Excel 大表格 | 分离，FAISS 只管向量 |
| **Chroma** | SQLite + Parquet | MySQL + 文件存储 | SQLite 管元数据，Parquet 管向量 |
| **Qdrant** | Segment + WAL | HBase 列族 | 分文件存储，Point ID 关联 |
| **Milvus** | LSM-tree + 对象存储 | Cassandra/TiKV | Segment 内按列存储 |
| **Weaviate** | 双 LSM-tree | LevelDB×2 | 向量和属性各一棵树 |

## 三、距离度量（怎么定义"相似"）

### 3.1 三种度量方式

| 度量 | 公式 | 范围 | 最适合 |
|------|------|------|--------|
| **余弦相似度** | cos(A,B) = A·B / (\|A\|·\|B\|) | [-1, 1] | 文本/语义（看方向，忽略长度） |
| **欧氏距离** | \|\|A-B\|\| = √Σ(Ai-Bi)² | [0, +∞) | 图像/数值（看空间绝对距离） |
| **点积** | A·B = Σ(Ai×Bi) | (-∞, +∞) | ML 内积（归一化后 = 余弦） |

**FAISS 实现：** IndexFlatIP（内积）+ normalize_L2（归一化）= 余弦相似度。

### 3.2 距离度量 vs 索引算法 vs 检索范式

度量只是定义了"怎么算相似"，但面对海量数据，逐个比较太慢了。这时候需要索引算法来加速搜索。同时，检索策略也不止向量一种。这三者是**不同层面**的概念，可以自由组合：

| 层次 | 概念 | 回答的问题 | 类比 |
|------|------|-----------|------|
| **距离度量** | 余弦/欧氏/点积 | 两个向量有多像？ | 距离尺子（量远近） |
| **索引算法** | Flat/HNSW/IVF | 如何快速找到最像的？ | 找路方法（怎么找） |
| **检索范式** | 向量/BM25/混合 | 用什么策略找？ | 搜索引擎（语义还是关键词） |

**度量决定"找什么"，算法决定"怎么找"。** 三者正交，常见组合：
- **余弦 + HNSW** = 最常见的组合
- **欧氏 + IVF+PQ** = 图像检索常见
- **余弦 + Flat** = 本项目当前方案（学习阶段最简单）

既然度量只解决了"怎么算分"，那面对百万级数据，怎么不逐个比较也能快速找到最近的？

## 四、索引算法详解（怎么快速搜索）

### 4.1 Flat（暴力搜索）

原理：把所有向量存下来，查询时逐一比较。

`数据库: [v1, v2, ..., v10000]` → 查询 q 逐个比较 → 取最近 K 个

- **时间复杂度**：O(n × d)
- **优点**：100% 召回率，结果最准
- **缺点**：数据量大时极慢（10万条 ≈ 几秒）
- **适用**：数据量 < 1万条

### 4.2 IVF（倒排索引）

原理：先把所有向量分成若干"桶"（聚类中心），查询时只搜附近的桶。

训练阶段：
1. 用 K-Means 把向量聚成 N 个簇（比如100个桶）
2. 每个向量分配到最近的簇

查询阶段：
1. 找到 query 最近的几个簇（nprobe=10，即搜10个桶）
2. 只在这10个桶里暴力搜索

- **时间复杂度**：O(n/N × d)，N 是桶数
- **召回率**：90-98%
- **优点**：比 Flat 快 10-100 倍
- **缺点**：需要训练，数据分布变化时需要重建
- **适用**：1万-100万数据，可接受微小精度损失

### 4.3 HNSW（分层小世界图）

原理：构建一个多层图结构，每层都是一个小世界网络。

```
底层（Layer 0）:  所有向量都在，互相连接
中间层（Layer 1）: 部分向量，稀疏连接
顶层（Layer 2）:  极少数向量，很稀疏
```

查询过程：
1. 从顶层的入口点开始
2. 每层贪心搜索，找到最近的节点
3. 下降到下一层，继续搜索
4. 在底层找到最终的 K 个最近邻

- **时间复杂度**：O(log n × d)
- **召回率**：95-99%
- **优点**：查询极快，不需要训练，支持动态插入
- **缺点**：内存占用大（每个向量要存多个邻居指针）
- **适用**：百万级数据，对召回和速度都有要求（**首选**）

### 4.4 PQ（乘积量化）

原理：把高维向量压缩成短编码，用查表代替计算。

压缩过程：
1. 把 512维 向量切成 32 个子空间，每个子空间 16 维
2. 每个子空间用 K-Means 聚成 256 个中心（1个字节表示）
3. 原始 512×4字节 = 2048字节 → 32字节（压缩 64 倍）

查询过程：
1. 预计算 query 与 256×32 个中心的距离表
2. 用查表快速估算距离

- **内存**：减少 90%+
- **速度**：极快（查表代替浮点运算）
- **召回**：70-90%（有精度损失）
- **适用**：内存受限、亿级数据

### 4.5 算法选择决策树

```
数据量多少？
├── < 1万     → Flat（精确，无需索引）
├── 1万-100万 → IVF 或 HNSW
│   ├── 需要动态增删 → HNSW
│   └── 内存紧张   → IVF + PQ
└── > 100万   → HNSW（首选）或 IVF+PQ（省内存）
```

索引算法解决了"怎么搜得快"，但检索策略本身也有讲究：只用向量够不够？要不要加上关键词？怎么组合效果最好？

## 五、检索范式（向量 vs 关键词 vs 混合）

### 5.1 三种范式对比

| 范式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **向量检索** | Embedding + 余弦相似度 | 懂语义："音箱"能匹配"扬声器" | 不懂精确匹配 |
| **BM25** | 分词 + TF-IDF变体 | 精确匹配："ZY-SP100"精准命中 | 不懂语义 |
| **混合检索** | 向量 + BM25 加权融合 | 兼顾语义和精确匹配 | 需调整权重 |

### 5.2 混合检索流程

```
Query ──→ 向量检索 Top-K*2 ──→ 分数归一化 ──┐
         (语义理解)                        ├→ 加权融合 → 排序 → Top-K
Query ──→ BM25 检索 Top-K*2 ─→ 分数归一化 ──┘
         (关键词精确)                   (0.7向量 + 0.3BM25)
```


### 5.3 为什么要做混合检索

向量检索和 BM25 各有明显的盲区，单独用任一种都会丢掉好结果：

向量检索的弱点

向量检索基于 Embedding 的语义相似度，它**理解意思但不认字**：

- ✅ “音箱价格” 能匹配 “音响多少钱”（语义相近）
- ❌ “音箱价格” 很难精确匹配文档标题中的“音箱”二字

结果是：语义泛化好，但分数差异小，很难拉开排名。

BM25 的弱点

BM25 是传统关键词检索，它**认字但不懂意思**：

- ✅ “音箱价格” 精确匹配文档中包含“音箱”“价格”的段落
- ❌ “音箱价格” 无法匹配 “智能音箱多少钱”（没命中关键词）

结果是：关键词精确，但语义为零。

混合检索让两者互补

Day 4 的测试数据直接说明了问题：

| 查询 | 纯向量 | 混合 | 原因 |
|------|--------|------|------|
| 音频规格 | 0.306 | **1.032** | BM25 精确命中“音频规格”关键词，混合后分数暴涨 |
| 音箱价格 | 0.548 | 0.623 | BM25 补充了关键词匹配 |
| 保修期 | 0.541 | 0.490 | BM25 无匹配，反而拖后腿 |

可以看到：向量检索擅长“理解意思”，BM25 擅长“精确匹配”，混合检索让两者各补短板。

> 最佳实践：向量 0.7 + BM25 0.3，先粗筛再精排。后面还需要 Reranker 做精排，两步配合才能拿到真正好的结果。

## 六、核心接口设计

### 6.1 VectorStore

```
add(vectors, texts, metadatas)   → FAISS 索引 + 文本存储

```

### 6.2 BM25Retriever

```
add_texts(texts, metadatas)     → 分词 + BM25 索引
search(query_text, top_k)       → List[SearchResult]
```

### 6.3 HybridRetriever

```
search(query_vector, query_text, top_k)
    → 1. 向量检索 Top-K*2
    → 2. BM25 检索 Top-K*2
    → 3. 分数归一化
    → 4. 加权融合 (0.7向量 + 0.3BM25)
    → 5. 排序取 Top-K
```

### 6.4 完整流程

```
添加知识库:
  文档 → 分段 (Day2) → Embedding (Day3) → FAISS + BM25 双索引

用户查询:
  问题 → Embedding → 向量检索 → 混合检索
                    → 关键词检索 → 
                              ↓
                          重排序
                              ↓
                          Top-K 结果
```

## 七、完整检索流程顺序

```
用户提问: "音箱的价格是多少？"
         │
         ▼
第1步  Embedding（Day3）
       "音箱的价格是多少？" → [0.12, -0.34, 0.56...]
       文本变成向量
         │
         ▼
第2步  用索引算法快速搜索（Flat/HNSW/IVF）
       "10万条数据里怎么不全扫也能找到最近的？"
       → 选了 Flat（暴力搜索）
         │
         ▼
第3步  选距离度量（余弦/欧氏/点积）
       "两个向量怎么算相似？"
       → 选了余弦（归一化 + 内积）
         │
         ▼
第4步  混合检索（向量 + BM25 融合）
       "语义匹配 + 关键词匹配 一起用"
       → 0.7向量 + 0.3BM25
         │
         ▼
第5步  重排序（Reranker）
       "粗筛结果再精排一遍"
         │
         ▼
       返回 Top-K 结果
```

流程跑通了，那实际效果怎么样？混合检索真的比纯向量好吗？

## 八、检索测试结果

### 8.1 纯向量 vs 混合检索对比

| 查询 | 纯向量最佳匹配 | 混合匹配 | 混合提升 |
|------|---------------|---------|---------|
| 音箱价格 | 0.548 | 0.623 | ✅ BM25 加持 |
| 保修期 | 0.541 | 0.490 | ⚠️ BM25 无匹配 |
| 音频规格 | 0.306 | 1.032 | ✅ 关键词强匹配 |

### 8.2 核心发现

- 向量检索：语义泛化好，但分数差异小
- BM25：关键词精确，但不懂语义
- 混合：两者互补，但需调整权重
- **最佳实践**：向量 0.7 + BM25 0.3，先粗筛再精排

本地跑通了，但如果要上生产，FAISS 只是冰山一角。市面上还有哪些向量库？各自适合什么场景？
