## 二、为什么需要评估

### 2.1 没有评估的问题

```
用户: "ZY-SP100 保修期多久？"
系统: "抱歉，知识库中没有找到保修信息..."
```

到底回答得好不好？不知道。
检索到的内容相关不相关？不知道。
换个参数效果会不会更好？不知道。

### 2.2 评估的价值

- 量化表现：用数字说话，不靠感觉
- 找出薄弱环节：知道哪里需要优化
- 对比不同配置：换参数后效果是变好还是变差
- 持续改进：建立"测试-优化-验证"的闭环

## 三、评估指标体系

### 3.1 检索质量

**Precision（检索精度）**：检索结果中有多少是相关的

```
检索返回 3 篇文档，其中 2 篇相关
Precision = 2/3 = 66.7%
```

**Recall（检索召回率）**：相关文档有多少被检索到了

```
知识库中有 3 篇相关文档，检索到了 2 篇
Recall = 2/3 = 66.7%
```

### 3.2 回答质量

**关键词命中率**：回答中是否包含期望的关键信息

```
期望关键词: ["899", "699", "元"]
回答: "ZY-SP100 售价 899 元，优惠价 699 元"
命中: 3/3 = 100%
```

**回答相关性**：LLM 评判回答与问题的匹配度（0-1 分）

### 3.3 综合得分

```
综合 = Precision x 0.2 + Recall x 0.2 + 关键词命中 x 0.3 + 回答相关性 x 0.3
```

权重说明：回答质量（0.6）比检索质量（0.4）更重要，因为用户最终看的是回答。

## 八、优化实验与对比

### 8.1 优化策略

| 优化项 | 优化前 | 优化后 | 预期效果 |
|--------|--------|--------|---------|
| chunk_size | 300 | 200 | 更小的块 = 更精准的检索 |
| Top-K | 3 | 5 | 更多候选 = 更好的召回 |
| System Prompt | 基础版 | 增加格式要求和超出范围处理 | 回答更简洁规范 |

核心思路：
- chunk 太大导致一个块包含多个主题，检索时容易匹配到无关内容 -> 缩小 chunk
- Top-K 太小导致相关文档被遗漏 -> 增加候选数
- Prompt 不够明确导致超出范围时回答啰嗦 -> 优化 Prompt

### 8.2 对比结果（LLM 评判）

| 指标 | 优化前 | 优化后 | 变化 |
|------|--------|--------|------|
| 回答质量（LLM 评判） | 84.00% | 85.33% | +1.33% |
| 综合得分 | 73.07% | 72.80% | -0.27% |

> 说明：早期使用关键词匹配评估，指标变化不明显。改用 LLM 评判后，能更准确地反映回答质量的真实变化。

### 8.3 逐题对比

**显著提升的题目：**

| 题目 | 优化前 | 优化后 | 提升 | 原因 |
|------|--------|--------|------|------|
| ZY-SP100 尺寸是多少？ | 0.75 | 0.92 | +0.17 | 更小的 chunk 精确命中了尺寸信息 |
| 买5台首发优惠价多少钱？ | 0.60 | 0.84 | +0.24 | 检索到了价格信息并正确计算 |
| ZY-SP100 保修期多久？ | 0.68 | 0.80 | +0.12 | 回答更准确，不再答非所问 |

**退化的题目：**

| 题目 | 优化前 | 优化后 | 退化 | 原因 |
|------|--------|--------|------|------|
| ZY-SP100 支持哪些音频格式？ | 0.33 | 0.00 | -0.33 | chunk 太小，音频规格信息被拆散 |
| ZY-SP100 保修期多久？ | 0.40 | 0.20 | -0.20 | 保修信息在更小的 chunk 中丢失 |

### 8.4 关键发现

1. **chunk_size 是把双刃剑**：缩小 chunk 提升了精准匹配（尺寸问题 +0.35），但也导致部分信息丢失（音频格式 -0.33）
2. **没有银弹**：单一参数调整无法同时提升所有题目
3. **数据质量是根本**：用户评价（ID 9）无论怎么调参数都无法提升，因为知识库中评价数据本身不完整
4. **Prompt 优化有效**：超出范围的问题（天气、保修期）回答更简洁了

### 8.5 进一步优化方向

| 方向 | 方法 | 预期效果 |
|------|------|---------|
| 混合 chunk_size | 不同文档用不同大小的 chunk | 兼顾精准和完整 |
| 知识图谱 | 提取实体关系，支持推理 | 推理类问题提升 |
| 查询改写 | 用 LLM 改写用户问题 | 检索命中率提升 |
| 多路召回 | 向量 + 关键词 + 全文检索 | 召回率提升 |
| 补充数据 | 添加保修政策、用户评价等 | 覆盖更多问题类型 |

## 九、生产级优化策略

### 9.1 核心区别

我们做的：手动调参数 -> 跑评估 -> 再调
生产项目的：自动化评估 -> 自动发现问题 -> 自动/半自动调整 -> 自动验证

### 9.2 查询改写（Query Rewriting）

用户的问题往往不是好的检索 query。生产项目先用 LLM 改写：

```
用户原始问题: "ZY-SP100 适合放卧室用吗？"

改写后:
  - 检索 query: "ZY-SP100 尺寸 体积 适用场景"
  - 推理 query: "ZY-SP100 的物理尺寸是否适合卧室摆放"

改写后检索命中率大幅提升
```

Dify 和 LlamaIndex 都有这个模块。这一招就能解决我们"尺寸问题"和"卧室问题"的退化。

### 9.3 多路召回（Multi-Path Retrieval）

不只用一种检索方式，而是多路并行：

```
用户问题
    |
    +-- 向量检索（语义匹配）    --> 结果集 A
    +-- 关键词检索（BM25）      --> 结果集 B
    +-- 全文检索（Elasticsearch）-> 结果集 C
    +-- 知识图谱查询            --> 结果集 D
    |
    v
  分数融合 + 去重 + Rerank
    |
    v
  最终结果
```

这就是为什么我们的 chunk_size 调小后音频格式问题退化了——如果有多路召回，即使向量检索没命中，关键词检索也能兜底。

### 9.4 自适应参数（Adaptive Parameters）

不同问题用不同参数，而不是全局统一：

```python
class AdaptiveRetriever:
    def retrieve(self, query):
        qtype = self.classify_query(query)

        if qtype == "fact":          # 事实性问题
            top_k = 3                # 精准匹配，少取
            chunk_size = 200         # 小 chunk
        elif qtype == "reasoning":   # 推理性问题
            top_k = 7                # 多取，给 LLM 更多上下文
            chunk_size = 400         # 大 chunk，保留完整语境
        elif qtype == "comparison":  # 对比类问题
            top_k = 10               # 需要多个产品的信息
            chunk_size = 300

        return self.search(query, top_k=top_k, chunk_size=chunk_size)
```

### 9.5 自动化评估流水线

```
每次代码变更（改 Prompt、换模型、调参数）
    |
    v
自动跑测试集（几百条问题）
    |
    v
对比前后指标
    |
    v
如果整体提升 -> 合并
如果部分退化 -> 分析原因，决定是否合并
```

Dify 有完整的 CI/CD + 评估流水线。LlamaIndex 有 EvaluationModule。

### 9.6 用户反馈闭环

```
用户提问
    |
    v
系统回答 + 显示"有帮助/没帮助"按钮
    |
    v
用户点击反馈
    |
    v
反馈数据 -> 优化检索/回答策略
```

这是最有效的优化方式——用真实用户数据驱动优化，而不是靠猜测。

### 9.7 缓存 + 预计算

```
常见问题预计算:
  "ZY-SP100 价格多少？" -> 预先检索好，直接返回

语义缓存:
  用户问"音箱多少钱" -> 和"ZY-SP100 价格多少"语义相似 -> 复用缓存结果
```

### 9.8 优化策略对比

| 策略 | 我们做的 | 生产项目做的 |
|------|---------|------------|
| 参数调整 | 手动调全局参数 | 根据问题类型自适应 |
| 检索方式 | 单路（向量+BM25） | 多路并行（向量+关键词+全文+图谱） |
| 查询处理 | 直接用原始问题 | 先改写/扩展再检索 |
| 评估方式 | 手动跑 15 条 | 自动化跑几百条 + CI/CD |
| 优化驱动 | 靠经验猜测 | 靠用户反馈数据 |
| 缓存 | 无 | 语义缓存 + 预计算 |

核心：生产项目的优化不是调参数，而是建立"发现问题 -> 定位原因 -> 自动修复"的闭环。
